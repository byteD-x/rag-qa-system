# 后端工程深入

## 概览

该项目的后端不是简单的“上传文档 + 调模型”，而是围绕企业知识库问答拆成了三条可证明的工程链路：

- 上传链路：前端分片直传对象存储，后端维护 upload session 与 ingest 作业。
- 检索链路：结构命中、FTS、向量检索融合，再做轻量 rerank。
- 回答链路：证据筛选、拒答判定、LLM 生成、成本与时延回传。

## 1. 上传与 ingest 的工程设计

### 为什么不用简单文件直传

企业文档往往体积大、网络不稳定、上传中断概率高。单次表单上传会把带宽、失败恢复和后端内存都压在 API 服务上。

### 采用的方案

- 浏览器通过预签名 URL 直接把分片上传到 MinIO/S3 兼容存储。
- `kb-service` 只负责：
  - 创建 upload session
  - 记录已上传 part
  - 完成 multipart merge
  - 创建文档记录和 ingest job
- `kb-worker` 异步处理解析、索引和 embedding。

### 工程价值

- 后端不承载大文件流量，服务更轻。
- 中断恢复依赖 `upload_id + uploaded_parts`，前端可断点续传。
- 上传与 ingest 解耦，便于后续演进到队列化或多 worker。

## 2. 文档生命周期设计

文档状态按以下阶段推进：

- `uploaded`
- `parsing_fast`
- `fast_index_ready`
- `hybrid_ready`
- `ready`
- `failed`

这套状态机的价值在于：

- 不要求“所有增强都完成后才能查询”。
- 用户能在 `fast_index_ready` 阶段先进行基础问答。
- `hybrid_ready` 和 `ready` 分别代表摘要向量和 chunk 向量就绪，状态含义清晰。

## 3. 检索链路设计

### 为什么做混合检索

纯向量检索对结构化标题和显式关键词不够敏感，纯全文检索又容易错过语义相近表达。企业知识库同时存在制度标题、章节结构和正文语义，因此必须做混合检索。

### 当前链路

1. `query rewrite`
2. `structure hit`
3. `FTS`
4. `vector retrieval`
5. `weighted RRF`
6. `lexical rerank`

### 工程取舍

- `query rewrite` 不走额外 LLM，避免把检索前置到更重的在线依赖上。
- RRF 保持信号融合简单稳定，避免复杂学习排序在零样本场景下难以维护。
- rerank 继续采用轻量 lexical overlap，而不是再引入重模型，兼顾效果与成本。

## 4. 证据化回答设计

网关统一输出：

- `answer_mode`
- `evidence_status`
- `grounding_score`
- `refusal_reason`
- `citations`
- `retrieval`
- `latency`
- `cost`
- `trace_id`

这样设计的价值：

- 前端能把“回答质量”和“检索质量”拆开显示。
- 面向排障时可以快速回答：是没检到、检到了但证据弱、还是生成端失败。
- 成本与时延被保存在同一条消息元数据里，方便后续审计与观测。

## 5. 显式初始化的工程化改造

原始实现把 migration 与对象存储桶创建放在服务启动里，适合快速 demo，但不适合工程化叙事。

本次改造后的原则：

- schema 初始化与服务启动职责分离
- `make init` 负责显式初始化
- 应用启动不再隐式改库

这让系统更像正式项目，而不是“应用顺手替你改环境”的样机。

## 6. 零数据基线的意义

删除 demo 数据和历史报告不是为了“看起来干净”，而是为了让项目边界更真实：

- 仓库本身不再依赖样例资产才能解释其能力
- 所有评测脚本必须显式传入输入
- 后续创建假数据时可按面试/简历场景定制，而不是被旧 demo 材料绑死

## 7. 资源隔离与一致性

当前后端把知识库、文档、上传会话和 ingest 作业都收敛到用户作用域下：

- 普通用户默认只能看到自己创建的资产
- 管理员可以做跨用户排障和维护

删除链路也从“先删对象存储再删数据库”改为“先完成数据库一致性删除，再做外部资源最佳努力回收”。这样做的核心取舍是：

- 优先保证系统对外可见状态一致
- 接受对象存储侧可能出现少量可回收孤儿资源
- 把清理失败降级为运维问题，而不是用户可见数据错乱
